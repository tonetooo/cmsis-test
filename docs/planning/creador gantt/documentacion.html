1. Vision General

Firmware para STM32F446RET6 (Cortex-M4) del sistema HERMES A1. Adquisicion autonomica de vibraciones y telemetria via 4G/LTE. Desarrollado por Antonio Avendano Sanhueza para LIND Engineering, Concepcion, Chile.

1.2 Principio de Operacion

El sistema opera en modo triggered acquisition: el ADXL355 genera una interrupcion (INT1) cuando la aceleracion supera un umbral configurable. El firmware lee el FIFO del acelerometro, almacena las lecturas en SD como archivos CSV, y las transmite a un servidor remoto via HTTP/4G.

El pipeline de datos es asincrono y concurrente: la adquisicion del sensor, la escritura en SD, la transmision por modem y la interfaz de usuario operan como tareas independientes de FreeRTOS, coordinadas mediante colas, mutex y event flags.

V1.2.0: La cola de upload es ahora un buffer circular en RAM (8 slots) que reemplaza la anterior cola basada en SD. El tracking de uploads se realiza con RTC backup registers (BKP0R..BKP19R) que sobreviven resets NRST. La adquisicion del sensor y la transmision por modem operan concurrentemente.

3.4 Flujo de Datos

V1.2.0 — Pipeline completo:

1. ADXL_INT1 (EXTI) → sensor_task (Motion detect)
2. sensor_task → acquire data → write CSV to SD (takes sd_mutex briefly)
3. sensor_task → upload_queue_push(filename) → EVT_FILE_READY
4. modem_task wakes → upload_queue_peek → Modem_UploadFile (3 attempts)
5. On success: queue_pop → .DONE marker → Modem_PowerOff → check next file
6. On NRST (from modem TCP close): queue already updated, .DONE already created
7. On reboot: file_task boot scan → backup register check → .DONE check → requeue

3.5 Tracking de Upload — Backup Registers RTC

El sistema HERMES A1 utiliza un mecanismo de doble capa para rastrear cuales archivos TRIG ya fueron subidos con exito:

Capa Primaria: RTC Backup Registers (BKP0R..BKP19R)

PropiedadDetalle
RegistrosBKP0R hasta BKP19R (STM32F446)
Tamaño32 bits por registro
Capacidad19 registros x 32 bits = 608 indices TRIG
BitmapBKP0R bits 0-31 = TRIG_001..032, BKP1R = TRIG_033..064, etc.
SupervivenciaSobreviven NRST, IWDG reset, software reset
LimpiezaSolo se limpian en Power-On Reset (POR)
EscrituraInstantanea (un solo registro de 32 bits)
FuncionesModem_MarkUploaded(idx), Modem_IsUploaded(idx)

Capa Fallback: .DONE Marker File en SD

PropiedadDetalle
FormatoTRIG_NNN.DONE (archivo vacio en FatFs)
Creacionf_open + f_close con HAL_Delay(50) post-write
VerificacionBoot scan abre archivo para lectura
LimitacionPuede no persistir si NRST golpea durante escritura NAND

Flujo en exito: Modem_CreateDoneMarker() → Modem_MarkUploaded() PRIMERO (instantaneo) → archivo .DONE (best-effort con delay de 50ms).

3.6 Boot Scan — Recuperacion post-reset

Al reiniciar (post-NRST), file_task ejecuta un escaneo del directorio SD para reconstruir la cola de upload:

1. Modem_BackupInit() — habilita RTC APB1 clock + DBP
2. f_mount + f_opendir en 0:/
3. Para cada TRIG_NNN.CSV encontrado:
   a. PRIMARY CHECK: Modem_IsUploaded(idx) → saltar si backup register set
   b. FALLBACK CHECK: .DONE file exists → saltar
   c. Si ninguno: agregar a pending_indices[]
4. Bubble sort pending_indices ascending
5. upload_queue_push para cada pendiente
6. Si hay pendientes: osEventFlagsSet(EVT_FILE_READY)

Este mecanismo garantiza que ningun TRIG se pierda, incluso despues de un NRST reset.

3.7 Root Cause del NRST — Modem Power Transient

PropiedadDetalle
Reset reason0x00000012 = PINRSTF + IWDGRSTF
Causa raizModem EC25 cierra TCP post-upload → actividad RF → pico de corriente en rail 3.3V compartido
AcoplamientoSe acopla al pin NRST (~40k ohm pull-up interno)

EvidenciaCONS_OK('Upload successful') NUNCA aparece en log → reset ocurre en ~5us

4.3 Justificacion de Heap_4

FreeRTOS ofrece 5 esquemas de heap. Se selecciono Heap_4 por: coalescencia (fusiona bloques libres adyacentes), alloc + free (soporta liberacion segura), first-fit (algoritmo determinista).

Riesgo residual: en operacion continua (24/7), la fragmentacion puede acumularse. Mitigacion: usar xPortGetFreeHeapSize() periodicamente para monitorear.

14. Rutas de Upload

HERMES A1 soporta dos rutas de upload para subir archivos CSV al servidor remoto:

Ruta 2 — Backend Flask (Principal)

El upload se realiza via HTTP POST a un servidor Flask accesible a traves de un Cloudflare Tunnel. El backend recibe el CSV, lo convierte a formato interno, y lo sube a Google Drive.
